Guidelines For Web Development

This guidlines a collection of conventions to website development suggested by many universities and promenant technology vendors. (i'll add sources as we read). But is there a standard? as far as i saw no. It turns out that web development is more flexible than that. and there are general guidelines that are adopted by most web developers which i will be going through here

Who is this for?

This is for users of all learning stages. It is a collection of basic guidelines makes the development much easier and enhance productivity whether you are a beginner or an expert.

Web development process can be broken down into 3 steps

  1. Directory structure
  2. Web page structure
  3. Web page presentation

Directory Structure

  • Folder name

  • File name

When i started this website the first thing i had to work with was the website directory. I quickly realized i needed to properly structure the website directory before it gets messy. A good directory structure is like code organization and code comments. It is not only important to understand the code but to know the content without extra clicks.

Folder name:

The root folders of a website typically has the following structure(note that all names are in lowercase
except for the README file).

Note also that I included several names for each folder. I found that htese are the most used (they are separated by a backslash):

  • js/scripts: Include the scripting files that take care of the web page behavior
  • css/styles: Include your CSS files that take care of the web page presentation/look
  • blog/posts/articles: Include the html files that take care of the web page structure
  • media/images/assets: Include the media resources of the website.
  • text/data: Includes the raw content of the web pages before any html structuring
  • index.html The main page
  • about.html The about page
  • logo.png The website logo
  • README.txt A text file that provide description of the directory (It is the equivalent of comments in codes). It is best to include one README file inside all root folders.

File name:

After naming and structuring the root folders i had to create my file naming scheme. I made my decisions based on the following guidelines:

  • Make it meaningful: Do not use arbitrary names like "myimage.jpg". This makes it hard to know what the image is later on and it also affects screen readers when they scan the name of the image.
  • Make it short: Longer names are harder to parse in some software and it also limit understanding. I try to make mine less than 25 characters.
  • Don't use spaces: If you use space it will be replaced by its hex value 20% when requested. Space signifies the end of a character and can halt the page from rendering properly if at all.
  • Don't use special characters: ~ ! @ # $ % ^ & * ( ) ` ; < > ? , [ ] { } ‘ “ with the exception are the hyphen and underscore character. why? because other characters have explicit programmatic purposes.
  • If using dates use YYYY-MM-DD or YY-MM-DD format.
  • Be consistent: with whatever style you decide to pursue.

Which to use underscore or hyphen?

  • Use hyphen instead of underscore. why? SEO support, visible when hyperlink is rendered in screen, and some SE do not recognize underscore.
  • Use all lowercase characters: This is to avoid operating systems compatibility and usability issues (i.e linux treat my-file and My-file as two different files but windows does not)
  • Use leading zeros: when naming sequential files use a leading zero (i.e file-01, file-001, etc).

Naming example 1: this-is-a-blog-post.html

Naming example 2: sunflower-200w-01.png

For a complete examples look up the directory of my website.

Web Page development

  • html and web page structure

  • CSS and web page presentation

After getting the directory structure figured out and you have created a naming plan that fits your needs (and that you should be consistent with). It is time to check how to develop a page.

html and web page structure

html is used to structure the raw content into (meaningful) sections. I When structuring a web page your users you should code the element it in a way that makes the code itself easy to read later that it is meaningfull normal users and users that use screen readers to navigate.

Best practice
  • Use as little markup as possible: this makes the page more readable and more meaningful (avoid using too much div element)
  • Make the elements meaningful: This means choosing an element that best describe the content it contain. (i.e using <p> for paragraphs and <li> for lists)
  
finding the and the right elements will benefit you and your users greatly. Just like we did with structuring the directory.
  • It will make the code more readable to you and your team.
  • It will be more meaningful and that will help users who use screen readers to navigate through the site.
  • It will improve the site SEO

I have a ready made template that i use for all the pages.
The most used html tags are <p> (paragraphs)

CSS and web page presentation

After getting the html page ready we get to the presentation section. several points to focus on when styling a page. like html structure it boils down to usability


Fonts: There are several good fonts that you can choose from but the best font for the web is Arial. followed by the "times roman".

Use a font that is available to all users: what this means is that you should prioritize font types that are pre installed in the users computers. you can chain as many fonts as you want starting with the most important.

Using the following property

font-type: arial, "times roman", sans-serif;

Make fonts readable.
Use fonts that are good for reading on the web. avoid art styled fonts for normal document flow even for titles.

Choose a font color that contrast with the background.

This is self explanatory. you should use colors that does not obfuscate the words
